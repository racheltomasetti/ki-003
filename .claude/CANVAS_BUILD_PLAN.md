## What the Canvas Is

The canvas is not a visualization of captures. Captures never appear on the canvas as nodes. The canvas is a thinking and synthesis workspace where the user distills their accumulated thinking into tangible visual artifacts — system diagrams, flowcharts, mind maps, process flows, free-form thinking diagrams. Whatever shape the thinking needs to take.

The project corpus — all captures and enrichments tagged to this project — is the invisible context the agent draws from. It never surfaces directly on the canvas. It powers the agent in the sidebar who helps the user synthesize and build diagrams.

The full loop:

```
Captures (mobile) → project corpus
       ↓ invisible context
Agent in sidebar
(reads corpus + brief + memory document + canvas state)
       ↓ conversation
User synthesizes thinking with agent
       ↓ expressed as
Visual diagram on canvas
(flowchart, system map, mind map, breakdown,
whatever shape the thinking needs to take)
```

## What the Canvas Is Not

- Not a visualization of the project corpus
- Not a place where captures appear as nodes
- Not auto-populated with project captures
- Not a viewer — it is a workspace

The canvas is the **Express** layer of the CODE loop (Capture, Organize, Distill, Express). Everything captured feeds into it invisibly. What comes out is tangible, actionable, and shareable.

## Canvas Library — tldraw

**Decision: tldraw. Not React Flow.**

React Flow is designed for structured node-edge graphs with defined node types. It works well for flowcharts but constrains mind maps and free-form thinking diagrams.

tldraw is designed for freeform canvas with shapes, connectors, text, and arrows. It accommodates system diagrams, flowcharts, mind maps, and free-form thinking all in the same workspace. The interaction quality for open-ended thinking matches the nature of the Ki canvas.

React 19 compatibility confirmed. Proceed with tldraw.

## The Sidebar Agent

The primary interface is a chat panel alongside the canvas. The user has a conversation with Ki. Ki reads the project corpus, project brief, memory document, conversation history, and existing canvas state. Based on the conversation Ki creates diagram elements on the canvas.

**Agent context layers in order:**

```
1. Memory document
   Who this person is — foundational context for all AI calls

2. Project brief
   What this project is about.
   The "Context for Claude" section is the key input.
   Generated by the existing generate-brief Edge Function.

3. All project captures with enrichments
   The full corpus for this project — the invisible knowledge base

4. Existing canvas state
   What nodes and edges already exist on this canvas.
   Agent reads this to extend and iterate rather than start fresh.

5. Conversation history
   What has been discussed in this canvas session.
   Persisted across sessions so the agent has continuity.
```

The agent must have access to the canvas state — existing nodes and edges — so it can add to, extend, and iterate on existing diagrams rather than starting from scratch every session. This is the reason the canvas state is stored as structured rows, not JSONB.

## Schema

Add these tables in a new migration. Do not modify existing migrations.

### canvas_nodes

```sql
create table public.canvas_nodes (
  id          uuid primary key default gen_random_uuid(),
  project_id  uuid not null references public.projects(id) on delete cascade,
  user_id     uuid not null references public.profiles(id) on delete cascade,
  node_id     text not null,
  type        text not null,
  content     text,
  position_x  float not null,
  position_y  float not null,
  width       float,
  height      float,
  style       jsonb,
  created_by  text not null default 'user'
              check (created_by in ('user', 'agent')),
  status      text not null default 'accepted'
              check (status in ('pending', 'accepted', 'rejected')),
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now(),
  unique (project_id, node_id)
);
```

### canvas_edges

```sql
create table public.canvas_edges (
  id          uuid primary key default gen_random_uuid(),

  edge_id     text not null,

  created_by  text not null default 'user'

              check (status in ('pending', 'accepted', 'rejected')),
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now(),
  unique (project_id, edge_id)

```

### canvas_conversations

```sql
create table public.canvas_conversations (
  id          uuid primary key default gen_random_uuid(),
  project_id  uuid not null references public.projects(id) on delete cascade,
  user_id     uuid not null references public.profiles(id) on delete cascade,
  role        text not null check (role in ('user', 'assistant')),
  content     text not null,
  created_at  timestamptz not null default now()

```

### RLS

Apply RLS to all three tables using the standard pattern:

```sql
alter table public.canvas_nodes enable row level security;
create policy "Users manage their own canvas nodes"
  on public.canvas_nodes for all
  using (auth.uid() = user_id);

alter table public.canvas_edges enable row level security;
create policy "Users manage their own canvas edges"
  on public.canvas_edges for all
  using (auth.uid() = user_id);

alter table public.canvas_conversations enable row level security;
create policy "Users manage their own canvas conversations"
  on public.canvas_conversations for all
  using (auth.uid() = user_id);
```

### updated_at triggers

Apply the existing `handle_updated_at` trigger function to `canvas_nodes` and `canvas_edges`:

```sql
create trigger set_canvas_nodes_updated_at
  before update on public.canvas_nodes
  for each row execute procedure public.handle_updated_at();

create trigger set_canvas_edges_updated_at
  before update on public.canvas_edges
  for each row execute procedure public.handle_updated_at();
```

## Agent Response Schema

The `canvas-agent` Edge Function returns structured canvas operations. Define and implement this schema before building the function.

```typescript
type CanvasOperation =
  | {
      type: 'create_shape'
      node_id: string
      shape_type: 'box' | 'circle' | 'diamond' | 'text' | 'arrow'
      content: string
      position: { x: number; y: number }
      size: { width: number; height: number }
      style?: Record<string, string>
    }
  | {
      type: 'create_connection'
      edge_id: string
      source_node_id: string
      target_node_id: string
      label?: string
    }
  | {
      type: 'create_text_block'
      node_id: string
      content: string        // markdown supported
      position: { x: number; y: number }
    }

type CanvasAgentResponse = {
  operations: CanvasOperation[]
  clear_pending: boolean     // when true, delete all existing status:'pending'
                             // nodes and edges before applying new operations
  summary: string            // what Ki created and why
  suggested_next?: string    // what to explore next
}
```

The app receives this response, applies the operations to the tldraw canvas, and persists each node and edge to the structured database rows. The agent creates. The user curates — they can edit or delete anything the agent produces.

## The Phantom Diagram Interaction

This is the core canvas interaction model. The user should rarely be manually creating individual shapes. The agent proposes full diagrams. The user curates.

**The flow:**

```
User describes what they want to visualize
→ Agent reads all five context layers
→ Agent returns CanvasAgentResponse
    with clear_pending: true and all operations as status: 'pending'
→ App clears all previous pending nodes and edges for this project
→ App writes new nodes and edges with status: 'pending'
→ tldraw renders them as phantoms
    (lighter opacity, dashed borders — visually distinct from permanent elements)
→ User reviews the proposal in the sidebar

User accepts
→ All pending nodes and edges → status: 'accepted'
→ tldraw renders them as solid, permanent elements

User requests changes
→ Sidebar conversation continues
→ Agent returns new response with clear_pending: true
→ New phantom replaces the old one
→ Repeat until accepted

User rejects
→ All pending nodes and edges deleted
→ Canvas returns to previous state
```

**Status rules:**
- User-drawn elements: `status: 'accepted'` immediately — real the moment they are drawn
- Agent-proposed elements: `status: 'pending'` until user accepts
- Only one pending proposal exists at a time — `clear_pending: true` enforces this
- `status: 'rejected'` is set transiently during the delete flow; rows are then hard-deleted

**tldraw rendering:**
Pending elements use tldraw's style system to render with reduced opacity and dashed borders. Accepted elements render normally. The visual distinction must be immediately unambiguous — the user must never be unsure what is a suggestion and what is permanent.

**Services additions for phantom flow:**
```
acceptPendingNodes(client, projectId)   // pending → accepted for all pending nodes
acceptPendingEdges(client, projectId)   // pending → accepted for all pending edges
clearPendingNodes(client, projectId)    // delete all pending nodes
clearPendingEdges(client, projectId)    // delete all pending edges
```

## What the Canvas Agent Reads

When invoked the `canvas-agent` Edge Function fetches all of the following before making the Claude Sonnet call:

```typescript
// 1. User memory document
const { memory_document } = await getProfile(client, userId)

// 2. Project brief
const { brief } = await getProject(client, projectId)

// 3. All project captures with enrichments
const captures = await getProjectCaptures(client, projectId)

// 4. Existing canvas state — nodes and edges
const nodes = await getCanvasNodes(client, projectId)
const edges = await getCanvasEdges(client, projectId)

// 5. Conversation history for this canvas
const history = await getCanvasConversations(client, projectId)
```

All of this goes into context before the Claude Sonnet call. The agent has complete knowledge of what exists on the canvas and can add to it, extend it, and connect new thinking to existing elements.

## Architecture Rules

These do not change.

- **Client injection.** No direct Supabase calls in canvas components. All data access via packages/services, client injected.
- **Canvas is web only.** Mobile is capture. Do not attempt to build canvas on mobile.
- **No hallucination.** Agent canvas content is grounded in the project corpus. The agent cannot reference captures that do not exist.
- **TypeScript strict.** No `any` types anywhere.
- **RLS always on.** Users only access their own canvas state.
- **Captures are immutable.** The canvas stores diagram content. It never modifies captures.body or any capture field.

---

## The Design Principle to Hold

The canvas is the Express layer. Captures are invisible context. What gets built on the canvas is new — a synthesis, a diagram, a system map — not a visualization of what was captured.

The agent helps create it. The user curates it. The thinking that comes out is theirs.

Do not auto-populate the canvas with project captures. The empty canvas that the user builds intentionally is the design. If the canvas auto-fills it becomes a viewer. Keep it a workspace.
